# 决策：降本降 Token 方案

**日期：** 2026-03-21

## 背景

随着项目进入高峰期，Token 消耗逐渐增加，需要制定降本方案。

## 当前 Token 消耗分析

### 会话统计
| 指标 | 数值 | 评估 |
|------|------|------|
| 上下文使用 | 87k / 256k (34%) | 正常范围 |
| 单次请求输入 | 1.5k tokens | 合理 |
| 单次请求输出 | 637 tokens | 合理 |
| 缓存命中率 | 98% | 优秀 |

### 上下文构成
| 来源 | 估算大小 | 占比 |
|------|----------|------|
| 系统提示 + 工具定义 | ~10k | 11% |
| 历史对话 | ~35k | 40% |
| 项目文件注入 | ~27k (27KB) | 31% |
| 当前请求 | ~15k | 18% |

## 决策点

### 1. 当前状况评估
- ✅ 34% 使用率还有 66% 余量
- ✅ 缓存命中率 98%，节省大量 Token
- ⚠️ 继续开发 2-3 小时后可能达到 70%+
- ⚠️ 文档文件（27KB）持续增长

### 2. 优化方案选择

| 方案 | 效果 | 操作 | 决策 |
|------|------|------|------|
| A. 压缩上下文 | 减少 40% | 执行 `/compact` 或新开会话 | ✅ 采用 |
| B. 文档瘦身 | 减少 30% | 大文档移到 archive/ 目录 | 暂缓 |
| C. 拆分会话 | 减少 50% | 每完成大功能新开会话 | ✅ 采用 |

### 3. 最终决策

**采用方案 A + C 组合：**

1. **立即执行：** 更新 PROJECT.md 完整记录项目状态
2. **新开会话：** 结束当前会话，新 AI 从 PROJECT.md 恢复上下文
3. **后续策略：**
   - 每完成一个大功能模块（如 Phase 3 完成）新开会话
   - 避免单次会话过长（建议控制在 2-3 小时内）
   - 重要决策及时记录到 query/ 目录

## 降本最佳实践

### 代码层面
- 减少单次提交文件数
- 避免重复读取相同文件
- 使用 `offset/limit` 读取大文件部分内容

### 对话层面
- 批量处理相关任务
- 避免频繁的小幅修改
- 使用 `/reasoning off`（已关闭）

### 工具层面
- `sessions_spawn` - 子代理分担任务
- `memory_search` - 避免重复查询
- 文件缓存 - 已自动启用（命中率 98%）
